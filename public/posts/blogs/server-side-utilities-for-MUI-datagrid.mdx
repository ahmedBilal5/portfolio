While I was replacing MUI `data-grid`s with server side lazy loaded `data-grid`s across a project at work, It quickly became obvious that I would need the appropriate functionality on the server-side to handle the data-grids filter, sort and pagination model, not to mention, start and end parameters for server-side lazy loading and group Keys for server side tree data.

The first instance of this functionality that I wrote was a class that performed all the work in memory at run-time. Here it is:

> NOTE: For brevity, I will not paste the whole implementation but just the core function of each class. Full implementations are linked at the bottom.
```typescript
export class DataGridUtils {
  /**
   * Generic method to filter and sort any array of objects
   */
  static processWithGridParams<T>(
    data: T[],
    filterModel?: DataGridFilterModelInput,
    //customFilters?: DataGridCustomFiltersInput[],
    sortModel?: DataGridSortItemInput[],
    paginationModel?: DataGridPaginationInput,
    start?: number,
    end?: number,
    treeData?: boolean,
    groupKeysInput?: GroupKeysInput,
  ): {
    data: T[];
    totalRecords: number;
  } {
    let result = [...data];
    // Apply filtering if provided
    if (filterModel && filterModel.items.length > 0) {
      result = this.filterData(result, filterModel, treeData, groupKeysInput);
    }

    // Apply quick filter if provided
    if (
      filterModel?.quickFilterValues &&
      filterModel.quickFilterValues.length > 0
    ) {
      result = this.applyQuickFilter(result, filterModel.quickFilterValues);
    }

    // Apply sorting if provided
    if (sortModel && sortModel.length > 0) {
      result = this.sortData(result, sortModel, treeData, groupKeysInput);
    }

    if (groupKeysInput) {
      result = this.filterBasedOnGroupKeys(result, groupKeysInput);
    }

    if (start !== undefined && end !== undefined) {
      return this.filterBasedOnStartAndEndIndices(result, start, end);
    }

    return this.paginate(
      result,
      paginationModel?.page,
      paginationModel?.pageSize,
    );
  }
}
```
The class above in totality contains all the methods for filtering, sorting, and all of the other data-grid configuration mentioned previously. The upside of this pure typescript implementation is that it can be flexibly be used regardless of what the data source is, since all the operations are happening in memory. The downside, and a major one, is that it requires the full set of data to be loaded into memory first, which sort of defeats the purpose of lazy-loading. This problem is exacerbated by the fact that the full set of data needs to be fetched on the server for each paginated/lazy loaded call to the server, which can be a lot. However, this class can be used conveniently when the volume of data is not massive.

Increasing volume of data in my project eventually led me to create a new implementation altogether in light of the drawback discussed: An implementation that does all the operations at the data source level, instead of memory. Since my data source was a Postgres relational database, the new implementation generates a sophisticated SQL query to select data from the database. I have leveraged the typeORM SelectQueryBuilder for much of this. Here is the new implementation:
```typescript
@Injectable()
export class DatagridQueryService {
  // Map of known relations for automatic joins
  private relationMap: Record<string, string[]> = {};

  /**
   * Allows you to set or override the relationMap at runtime.
   * @param map A dictionary of relation names to aliases
   *            e.g. { schema: 'schema', dataSource: 'dataSource' }
   */
  setRelationMap(map: Record<string, string[]>) {
    this.relationMap = map;
  }

  buildQuery<T>(
    repo: Repository<T>,
    alias: string,
    params: ZdpCommonDTOs.DataGridParams,
    forceJoinsMap: Map<string, string[]> = new Map(),
  ): SelectQueryBuilder<T> {
    const qb = repo.createQueryBuilder(alias);

    // Ensure joins exist if filter/sort fields reference relations
    this.ensureJoins(qb, alias, params, forceJoinsMap);

    // 1. Filters
    if (params.filterModel?.items?.length) {
      const logic =
        params.filterModel.logicOperator === GridLogicOperator.or
          ? 'OR'
          : 'AND';
      const conditions: string[] = [];
      const values: Record<string, any> = {};

      params.filterModel.items.forEach((f, idx) => {
        const { condition, params: condParams } = this.buildFilterCondition(
          alias,
          f,
          `val${idx}`,
        );
        conditions.push(condition);
        Object.assign(values, condParams);
      });

      qb.andWhere(conditions.join(` ${logic} `), values);
    }

    // 2. Custom filters
    params.customFilters?.forEach((cf, idz) => {
      const logic = cf.logicOperator === GridLogicOperator.or ? 'OR' : 'AND';
      const conditions: string[] = [];
      const values: Record<string, any> = {};

      cf.items.forEach((f, idx) => {
        const { condition, params: condParams } = this.buildFilterCondition(
          alias,
          f,
          `cf${idz}${idx}`,
        );
        conditions.push(condition);
        Object.assign(values, condParams);
      });

      qb.andWhere(conditions.join(` ${logic} `), values);
    });

    // 3. Sorting
    params.sortModel?.forEach((s) => {
      const { targetAlias, column } = this.parseField(s.field, alias);
      qb.addOrderBy(
        `${targetAlias}.${column}`,
        s.sort.toUpperCase() as 'ASC' | 'DESC',
      );
    });

    // 4. Pagination / lazy loading
    if (
      params.paginationModel?.pageSize &&
      params.paginationModel.pageSize > 0
    ) {
      qb.take(params.paginationModel.pageSize);
      qb.skip(params.paginationModel.page * params.paginationModel.pageSize);
    } else if (params.start !== undefined && params.end !== undefined) {
      qb.skip(params.start);
      qb.take(params.end - params.start);
    }

    return qb;
  }
``` 
This class delivers perfectly correct results, just as the previous class. However, it eliminates the problem of fetching everything into memory first, and instead will generate a query which will fetch the final set of results right from the source. This is a major improvement. However, it has been tested to work only with a Postgres database table.

#### Approach A: In-memory filtering


***Pros***

- Flexible to use

- Plain Typescript; easier to extend and refactor
  

***Cons***

- Fetches everything into memory; not true lazy loading
---

  

#### Approach B: Data source level filtering


***Pros***

- Fetches filtered data right from source as lazy loading should be ideally

***Cons***

- Requires a data source specific implementation; in my case SQL (Postgres)

- Harder to write and extend
---

Keeping everything in view, the second implementation must be considered to be the better one since it shifts the filtering operations to the data source level and thus, keeps the point of paginated loading alive.

I have focused this article on discussing the two approaches at a high level with respect to their pros and cons and have not dissected the implementations I attached. That is something I will leave up to you, the reader!
